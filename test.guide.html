Section 01

Why Write Tests?

Code without tests is like driving without a seat belt. The faster you go, the more the risk rises.

😰

Code Without Tests — Risks

  • Every new feature can break old functionality
  • Regression bugs — fix one place, break another
  • Fear of refactoring — you hesitate to change code
  • Production bugs — users find them
  • Manual testing takes time — checking everything by hand every time
🛡️

Code With Tests — Benefits

  • Regression protection — old features stay safe
  • Living documentation — tests show what the code does
  • Confident refactoring — tests back you up
  • Faster development — bugs are caught earlier
  • Team collaboration — new developers understand the code faster
⚠ Statistics

Finding a bug in production is 100x more expensive than catching it during development. Early testing saves more time than hours of debugging later.

// Test Flow — Comparison
WITHOUT TESTS
Code → Deploy → 🐛 Bug → Hotfix → 😰 Stress

WITH TESTS
Code → Test → ✅ Pass → Deploy → 😊 Confidence
Code → Test → ❌ Fail → Fix → ✅ Pass
Section 02

Test Types — Pyramid

Tests are divided into three main categories. Each has its own speed, cost, and scope.

// Test Pyramid
UNIT TESTS~70%
Many · Fastest · Cheapest
INTEGRATION~20%
Some · Balanced · Reliable
E2E TESTS~10%
Few · Slowest · Most expensive
Why this shape?

The lower a test sits in the pyramid, the cheaper and faster it is to run. Keep most of your coverage in the base, add fewer integration tests, and reserve end-to-end tests for critical user journeys.

Most commonly used ⚡ Very fast

Unit tests test a function or component in isolation . They do not depend on external dependencies (API, database). Each test runs independently.

utils/math.test.js
unit test 
// ✅ Unit test — single function
function add(a, b) {
  return a + b;
}

test('The add function returns 2 + 3 = 5', () => {
  const result = add(2, 3);
  expect(result).toBe(5);
});

// ✅ Unit test — React component
test('Button renders correctly', () => {
  render(<Button label="Click me" />);
  expect(screen.getByText('Click me'))
    .toBeInTheDocument();
});
Medium complexity 🏃 Fast

Integration tests verify several components working together . They check flows such as form submission and the result of an API call.

LoginForm.test.jsx
integration test 
// ✅ Integration test — form submission flow
test('shows a welcome message after submission', async () => {
  render(<LoginForm />);

  // Enter email
  await userEvent.type(
    screen.getByLabelText('Email'),
    'test@mail.com'
  );

  // Click the button
  await userEvent.click(
    screen.getByRole('button', { name: 'Sign in' })
  );

  // Check the result
  expect(screen.getByText('Welcome!'))
    .toBeInTheDocument();
});
Fewest 🐢 Slow

E2E (End-to-End) tests verify the full user flow in a real browser. Tools like Cypress and Playwright are used. They are the most expensive, but also the most realistic tests.

e2e/checkout.spec.ts
playwright 
// ✅ E2E test — full shopping flow
test('the user buys a product and pays', async ({ page }) => {
  await page.goto('/products');

  // Select a product
  await page.click('[data-testid="product-1"]');
  await page.click('text=Add to cart');

  // Payment
  await page.goto('/checkout');
  await page.fill('#card-number', '4111111111111111');
  await page.click('text=Pay');

  // Success message
  await expect(page.locator('text=Your order has been received'))
    .toBeVisible();
});
Feature Unit Integration E2E
What does it test? Single function Between components Full user flow
Speed ⚡⚡⚡ Very fast ⚡⚡ Fast ⚡ Slow
Cost Cheap Medium Expensive
Tool Jest, Vitest Jest + RTL Cypress, Playwright
Share in the pyramid ~70% ~20% ~10%
💡 Golden Rule

The base of the pyramid should be unit tests, the middle integration tests, and the top E2E tests. Doing the opposite is called the "ice cream cone anti-pattern."

Section 04

Jest Matchers — expect() Methods

Matchers are used to check the result. expect(value).matcher(expected) — simple, but powerful syntax.

.toBe()
Primitive value comparison (===). Use it for numbers, strings, and booleans.
.toEqual()
Deep comparison for objects/arrays. Use toBe only for primitive values.
.toBeTruthy()
Truthy value check. Anything that evaluates to true in an if condition.
.toBeFalsy()
Falsy value check. null, undefined, 0, false, "".
.toContain()
Searches for an element in an array. Also for a substring inside a string.
.toThrow()
Checks that a function throws an error. Be sure to wrap the function.
.toHaveLength()
Checks the length of an array or string.
.not
Opposite. .not.toBe(3) — not equal to 3.
matchers.test.js
javascript 
// Primitive comparison — toBe
expect(2 + 2).toBe(4);
expect('hello').toBe('hello');

// Object comparison — use toEqual!
expect({ name: 'Ali' }).toEqual({ name: 'Ali' });  // ✅
// expect({ name: 'Ali' }).toBe({ name: 'Ali' })  ❌ DOES NOT WORK

// Truthiness
expect('hello').toBeTruthy();
expect(0).toBeFalsy();
expect(null).toBeNull();
expect(undefined).toBeUndefined();

// Array
expect([1, 2, 3]).toContain(2);
expect([1, 2, 3]).toHaveLength(3);

// NOT — opposite
expect(5).not.toBe(3);
expect([1, 2]).not.toContain(5);

// Error — the function must be wrapped!
expect(() => {
  throw new Error('An error occurred!');
}).toThrow('An error occurred!');
⚠ toBe vs toEqual

toBe performs reference comparison (===). Even if two different objects have the same values false returns. toEqual performs deep comparison — it checks values, not the reference. For objects/arrays, always toEqual use it!

Section 05

Jest Mock — Simulate Dependencies

We replace external dependencies such as APIs, databases, and timers with fake versions during tests. This makes tests fast, reliable, and deterministic.

🌐

Why Do We Need Mocks?

  • API calls — no real server needed
  • External services — payments, sending email
  • Timers — setTimeout/setInterval
  • Deterministic — same result every time
  • Fast — no network delay
🎭

Types of Mocks

  • jest.fn() — creates a fake function
  • jest.mock() — mocks the entire module
  • jest.spyOn() — spies on an existing function
  • mockResolvedValue() — async result
  • mockRejectedValue() — simulates an error
callback.test.js 
test('callback is called correctly', () => {
  // Create a fake function
  const mockFn = jest.fn();

  // Use
  mockFn('hello');
  mockFn('world');

  // Assert
  expect(mockFn).toHaveBeenCalled();
  expect(mockFn).toHaveBeenCalledTimes(2);
  expect(mockFn).toHaveBeenCalledWith('hello');
});

test('returns a fake value', () => {
  const mockFn = jest.fn().mockReturnValue(42);
  expect(mockFn()).toBe(42);
});
api.test.js 
import { fetchUsers } from './api';

// Mock the global fetch
global.fetch = jest.fn();

test('fetches users', async () => {
  const mockUsers = [
    { id: 1, name: 'Ali' },
    { id: 2, name: 'Vali' }
  ];

  // define the fetch response
  fetch.mockResolvedValue({
    json: () => Promise.resolve(mockUsers)
  });

  const users = await fetchUsers();

  expect(fetch).toHaveBeenCalledWith('/api/users');
  expect(users).toEqual(mockUsers);
  expect(users).toHaveLength(2);
});

test('handles the error case', async () => {
  fetch.mockRejectedValue(new Error('Network error'));

  await expect(fetchUsers())
    .rejects.toThrow('Network error');
});
module-mock.test.js 
// Mock the whole module
jest.mock('./emailService');

import { sendEmail } from './emailService';

test('an email is sent on sign-up', async () => {
  sendEmail.mockResolvedValue({ sent: true });

  await registerUser({ email: 'test@mail.com' });

  expect(sendEmail).toHaveBeenCalledWith(
    expect.objectContaining({
      to: 'test@mail.com'
    })
  );
});

// spyOn — observe an existing method
test('console.error is called', () => {
  const spy = jest.spyOn(console, 'error')
    .mockImplementation(() => {});

  myBrokenFunction();

  expect(spy).toHaveBeenCalled();
  spy.mockRestore();
});
✅ Mock Rule

Only external dependencies mock APIs, databases, email services, and third-party libraries. Never mock the code you are actually testing — it makes the test meaningless.

Section 06

React Testing Library

RTL philosophy is simple: test components the way the user sees them. Behavior matters, not implementation details.

// RTL Philosophy
"The more your tests resemble the way your software is used, the more confidence they can give you."
— Kent C. Dodds (RTL author)
Feature Enzyme (old) RTL (modern) ✅
Approach Implementation details User behavior
Type of render Shallow render Full render (DOM)
Finding elements className, state role, text, label
Refactoring Tests break ❌ Tests pass ✅
Accessibility Weak support Strong support
💡 Why did RTL win?

Enzyme componentn how it worked. RTL tests the user's what they see . When you refactor, RTL tests do not break — because if the user interface has not changed, the tests still pass.

Greeting.test.jsx 
import { render, screen } from '@testing-library/react';
import Greeting from './Greeting';

test('displays the name correctly', () => {
  // 1. Render it
  render(<Greeting name="Ali" />);

  // 2. Find it on the screen
  const heading = screen.getByText('Hello, Ali!');

  // 3. Assert
  expect(heading).toBeInTheDocument();
});

test('shows a welcome message', () => {
  render(<Greeting name="Vali" />);
  expect(screen.getByText('Welcome!'))
    .toBeInTheDocument();
});
Writing tests with RTL has 4 steps:
1
render() — Komponenti virtual DOM-a render et
2

3
userEvent — Interact like a user
4
expect() — Verify the result
Section 07

screen Queries — Finding Elements

RTL provides various query methods for finding elements. Each has different behavior.

getBy...

Returns the element when found. When not found — throws an error. Most commonly used.

  • 0 found → ❌ Error
  • 1 found → ✅ Element
  • 1+ found → ❌ Error

    • queryBy...

    Returns the element when found. When not found — returns null. For "does not exist" checks.

  • 0 found → null
  • 1 found → ✅ Element
  • 1+ found → ❌ Error

    • findBy...

    Async — waits for the element. For elements that appear after an API response. Default waits 1000ms.

  • 0 found → ❌ Error (timeout)
  • 1 found → ✅ Promise
  • await use with

    • Query Priority Order

    🥇 1
    getByRole — button, heading, textbox, checkbox, link... Accessibility-friendly. Best choice.
    🥈 2
    getByLabelText — For form elements. Inputs tied to labels.
    🥉 3
    getByPlaceholderText — Find by input placeholder.
    4
    getByText — Find by visible text. Buttons, headings, paragraphs.
    5
    getByTestId — With data-testid attribute. Last resort — only when there is no other way.
    Navigation.test.jsx
    getByRole example 
    test('finds elements by role', () => {
  render(<Navigation />);

  // Heading
  screen.getByRole('heading', { name: 'My Site' });

  // Button
  screen.getByRole('button', { name: 'Log in' });

  // Link
  screen.getByRole('link', { name: 'About' });

  // Checkbox
  screen.getByRole('checkbox');
});

test('error message is not shown initially', () => {
  render(<LoginForm />);

  // queryBy — returns null, does not throw ✅
  const error = screen.queryByText('Error!');
  expect(error).not.toBeInTheDocument();
});
    Section 08

    userEvent — User Interactions

    userEvent behaves like a real user. Keyboard presses, focus, clicks, typing — all are simulated.

    💡 fireEvent vs userEvent

    fireEvent sends DOM events directly. userEvent follows the real browser sequence — focus → keydown → input → keyup → blur. It is more precise and reliable. Prefer userEvent for user-facing interactions.

    Counter.test.jsx 
    import { render, screen } from '@testing-library/react';
import userEvent from '@testing-library/user-event';
import Counter from './Counter';

describe('Counter component', () => {
  const user = userEvent.setup();

  test('shows initial value 0', () => {
    render(<Counter />);
    expect(screen.getByText('Counter: 0'))
      .toBeInTheDocument();
  });

  test('The increase button increments the value', async () => {
    render(<Counter />);
    const incrementBtn = screen.getByRole('button',
      { name: /Increase/i });

    await user.click(incrementBtn);
    await user.click(incrementBtn);
    await user.click(incrementBtn);

    expect(screen.getByText('Counter: 3'))
      .toBeInTheDocument();
  });

  test('The reset button works', async () => {
    render(<Counter />);
    const incrementBtn = screen.getByRole('button', { name: /Increase/i });
    const resetBtn = screen.getByRole('button', { name: /Reset/i });

    await user.click(incrementBtn);
    await user.click(incrementBtn);
    expect(screen.getByText('Counter: 2')).toBeInTheDocument();

    await user.click(resetBtn);
    expect(screen.getByText('Counter: 0')).toBeInTheDocument();
  });
});
    LoginForm.test.jsx 
    describe('LoginForm', () => {
  const mockSubmit = jest.fn();
  const user = userEvent.setup();

  beforeEach(() => {
    mockSubmit.mockClear();
    render(<LoginForm onSubmit={mockSubmit} />);
  });

  test('shows an error for a short password', async () => {

    expect(screen.getByRole('alert'))

  });

  test('submits the form with valid data', async () => {
    await user.type(screen.getByLabelText('Email'), 'test@mail.com');
    await user.type(screen.getByLabelText('Password'), 'password123');

      email: 'test@mail.com',

    TodoApp.test.jsx 
    describe('TodoApp', () => {
  const user = userEvent.setup();

  beforeEach(() => render(<TodoApp />));

    Async Testing — Components with APIs

    In real applications, components load data from APIs. We test these scenarios findBy and waitFor with

    UserList.test.jsx
    async test 
    beforeEach(() => {
  global.fetch = jest.fn();
});

test('loads users', async () => {
  fetch.mockResolvedValue({
    json: () => Promise.resolve([
      { id: 1, name: 'Ali' },
      { id: 2, name: 'Vali' }
    ])
  });

  render(<UserList />);

  // Loading is shown
  expect(screen.getByText('Loading...')).toBeInTheDocument();

  // Wait for the API response — findBy waits asynchronously
  expect(await screen.findByText('Ali')).toBeInTheDocument();
  expect(screen.getByText('Vali')).toBeInTheDocument();

  // Loading is no longer present
  expect(screen.queryByText('Loading...'))
    .not.toBeInTheDocument();
});

test('shows the error state', async () => {
  fetch.mockRejectedValue(new Error('Network'));
  render(<UserList />);

  expect(await screen.findByRole('alert'))
    .toHaveTextContent('An error occurred!');
});
    💡 findBy = getBy + waitFor

    findBy automatically waits for DOM updates, which makes it ideal for async data from APIs. The default timeout is 1000ms. getBy fails immediately if the element is not there yet.

    Best Practices & AAA Pattern

    Good testing is not just a technical skill; it is a mindset. Following these rules makes your tests maintainable, readable, and reliable.

    A
    Arrange
    Arrange — render the component, define the variables
    A
    Act
    Act — click, type, API call
    A
    Assert
    Assert — verify the expected result
    aaa-pattern.test.jsx
    AAA pattern 
    test('adding a todo updates the list', async () => {
  const user = userEvent.setup();

  // ── ARRANGE — prepare ─────────────────
  render(<TodoApp />);
  const input = screen.getByRole('textbox', { name: 'New task' });
  const addBtn = screen.getByRole('button', { name: 'Add' });

  // ── ACT — act ──────────────────
  await user.type(input, 'Learn React');
  await user.click(addBtn);

  // ── ASSERT — verify ────────────────────
  expect(screen.getByText('Learn React')).toBeInTheDocument();
  expect(input).toHaveValue('');  // input cleared
});

    ✅ Best Practices

    • Test user behavior
    • Prioritize getByRole
    • Write accessible tests
    • Mock only external dependencies
    • Write clear, descriptive test names
    • Arrange → Act → Assert (AAA)
    • Each test should verify one behavior
    • Reduce repetition with beforeEach

    ❌ Bad Practices

    • Testing implementation details
    • Using container.querySelector
    • Overrelying on snapshots
    • Mocking everything
    • Naming tests "test1", "test2"
    • Checking too many things in one test
    • Mocking the code you are testing
    • Forgetting async/await

    Section 11

    Knowledge Check

    Test what you learned. Strengthen your knowledge by answering each question.

    1. Which matcher should be used to compare objects?
    ✅ Correct! toEqual() performs deep comparison. toBe() performs reference comparison and does not work for objects.
    ❌ Wrong. toEqual() is the correct answer — it deeply compares object values.
    2. Which query returns null when an element is not found (without throwing)?
    ✅ Correct! queryBy returns null when the element is not found. It is used for "this element does not exist" checks.
    ❌ Wrong. queryByText() is the correct answer — it returns null when not found and does not throw.
    3. Which query has the highest priority in RTL?
    ✅ Correct! getByRole() is the highest priority. It is accessibility-friendly and checks how the element is seen by the user.
    ❌ Wrong. getByRole() is the correct answer — it is RTL's recommended first choice.
    4. Which Jest method is used to mock an API call?
    ✅ Correct! jest.fn() creates a fake function, .mockResolvedValue() while it defines the async response.
    ❌ Wrong. jest.fn().mockResolvedValue(data) is the correct combination.
    5. What do the "A" letters in the AAA pattern stand for?
    ✅ Correct! Arrange → Act → Assert. This pattern makes tests structured and readable.
    ❌ Wrong. AAA = Arrange → Act → Assert.
